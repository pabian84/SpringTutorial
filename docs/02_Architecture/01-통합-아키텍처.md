## 2. 인증/보안 아키텍처
본 프로젝트는 JWT 탈취 위험을 최소화하고 사용자 경험을 극대화하기 위해 이중 쿠키 및 세션 트래킹 하이브리드 방식을 사용합니다.

### 2.1 이중 쿠키 전략 (Dual-Cookie)
* **`accessToken` 쿠키 (HttpOnly, Session)**: XSS를 방어하며 브라우저 종료 시 삭제되는 단기 인증 토큰.
* **`deviceId` 쿠키 (HttpOnly, 1년 만료)**: 기기 고유 식별자. 브라우저 종료 후 재접속 시 서버의 `user_sessions` 테이블과 대조하여 기존 세션을 복구(Refresh)하는 데 사용됨.

### 2.2 세션 동기화 (Tab Sync)
여러 탭을 띄워둔 상태에서 한 탭에서 로그인/로그아웃 발생 시, `BroadcastChannel` (또는 `storage` 이벤트) 기반으로 다른 모든 탭의 인증 상태를 실시간으로 일치시킵니다.

## 3. 웹소켓 아키텍처 (Real-Time)
웹소켓은 인증된 사용자만 접근할 수 있도록 핸드셰이크 과정에서 `accessToken`을 검증합니다.
* **기능**: 채팅 브로드캐스트, 서버 모니터링, **다른 탭/기기 강제 로그아웃 알림**.
* **제어 흐름 (실시간 로그아웃)**: 
    1.  **API 요청**: 사용자가 다른 기기에서 강제 로그아웃 버튼을 누릅니다 (HTTP POST/DELETE).
    2.  **DB 세션 삭제**: 백엔드의 `SessionService`가 `user_sessions` 테이블에서 해당 세션 정보를 삭제합니다.
    3.  **웹소켓 강제 종료**: `SessionService`는 메모리상의 `webSocketSessionsMap`에서 해당 사용자의 소켓 세션을 찾아 **Close Code 4001**을 담아 연결을 즉시 끊어버립니다.
    4.  **클라이언트 리다이렉트**: 클라이언트(`WebSocketProvider`)는 `4001` 코드를 감지하는 즉시 로컬 상태를 초기화하고 로그인 페이지로 강제 이동합니다.

이 설계는 **'REVOKED(취소된)' 세션이 웹소켓을 통해 실시간 데이터를 계속 수신하는 보안 취약점**을 근본적으로 해결합니다.
